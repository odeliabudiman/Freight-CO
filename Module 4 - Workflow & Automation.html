② Written Step-by-Step Description
01
Trigger — Webhook or Schedule
n8n: Webhook Node + Cron Schedule Node
Two triggers feed the same workflow. Webhook: the driver mobile app POSTs to /webhook/delivery-update on every status change, firing the workflow in real-time. Cron: runs every 15 minutes as a safety net to catch silent failures — drivers who went offline without submitting a final status update.
02
Fetch In-Transit Deliveries
n8n: Supabase Node (SELECT)
Queries the Supabase deliveries table for all rows where status = 'in_transit' and actual_time IS NULL. Returns each delivery as a separate n8n item so downstream nodes process them in parallel.
-- Supabase filter
status = 'in_transit'
actual_time IS NULL
03
Compute Delay Minutes
n8n: Code Node (JavaScript)
For each delivery, calculates how overdue it is using the current server time vs the scheduled_time field. Outputs a computed_delay value in minutes, added as a new field on the item.
const delay = (Date.now()
  - new Date(scheduled_time))
  / 60000;
return { ...item, computed_delay: delay }
04
Gate 1 — Threshold Check (≥ 45 min)
n8n: IF Node
Filters items to only those where computed_delay ≥ 45. Items that pass continue down the true branch. Items below the threshold are dropped silently — no alert, no noise. This is the primary business rule gate from the PRD.
05
Gate 2 — Deduplication Check
n8n: Supabase Node + IF Node
Queries the alerts table: has an alert already been sent for this delivery_id within the last 2 hours? If yes — skip. This prevents coordinators from being spammed on every cron tick. The 2-hour window is configurable.
-- Dedup query
delivery_id = {{delivery_id}}
alert_sent_at > now() - interval '2h'
06
Lookup Hub Coordinator Contact
n8n: Supabase Node (SELECT)
Fetches the coordinator's name, phone number, and email from a hub_contacts table using the delivery's hub field (North / South / East). This keeps contacts editable by ops without touching the workflow.
07
Send SMS Alert via Twilio
n8n: Twilio Node
Sends a concise SMS to the hub coordinator. Message includes driver name, hub, delivery ID, and minutes late. Character limit kept to 1 SMS segment (<160 chars) to control cost.
// SMS body template
⚠️ LATE ALERT [{{hub}}]
Driver: {{driver_name}}
ID: {{delivery_id}}
{{computed_delay}}min overdue
08
Send Email Alert via SendGrid
n8n: SendGrid Node
Sends a richer HTML email with full delivery context: driver name, hub, scheduled vs actual time, delay summary, and a direct link to the Retool dashboard filtered to that driver. SendGrid free tier covers 100 emails/day — sufficient for 200 drivers.
09
Escalation Gate — ≥ 90 Minutes
n8n: IF Node → Twilio + SendGrid
A second IF node checks if computed_delay ≥ 90. If true, fires a separate SMS + email to the Operations Manager (not just the hub coordinator). This escalation path runs in parallel with the coordinator alert — both receive notifications simultaneously.
10
Log Alert & Mark Processed
n8n: Supabase Node (INSERT + UPDATE)
Two final writes: (1) INSERT a row into the alerts table with delivery_id, delay value, channel, and timestamp. (2) UPDATE the delivery row with alert_sent_at = now(). This feeds the dedup gate on subsequent runs and enables audit history for the dashboard.
